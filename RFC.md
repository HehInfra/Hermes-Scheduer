## 1. 背景与问题

### 1.1 现状
- 我们使用 vLLM / SGLang 部署 LLM 推理服务
- 单实例无法支撑高 QPS，需要多实例水平扩展

### 1.2 核心痛点
| 痛点 | 描述 |
|------|------|
| 负载均衡无效 | 传统 LB 不理解 LLM 特性，导致缓存命中率低 |
| 引擎碎片化 | vLLM 和 SGLang 接口、行为不一致 |
| 运维黑盒 | 缺乏针对推理场景的监控指标 |

## 2. 项目目标

### 2.1 核心目标
- 实现 LLM-aware 的路由调度器
- 统一 vLLM / SGLang 的 API 抽象
- 提供生产可用的可观测性

### 2.2 非目标
- 不替换底层推理引擎
- 不实现模型推理本身

## 3. 输入输出与系统边界

### 3.1 上边界：对外暴露的接口

- 暴露 OpenAI 兼容的 API（`/v1/chat/completions`、`/v1/completions`、`/v1/models`）
- 支持流式响应（SSE）
- 提供 `/health` 健康检查和 `/metrics` Prometheus 指标端点

### 3.2 下边界：接入的推理引擎

- 支持 vLLM 实例作为后端
- 支持 SGLang 实例作为后端
- 支持混合部署（同时接入两种引擎）

### 3.3 边界承诺

| 方向 | 做什么 | 不做什么 |
|------|--------|---------|
| 对上 | 提供 OpenAI 格式 API，透传标准参数 | 不做请求内容深度校验，不实现 OpenAI 高级功能校验 |
| 对下 | 健康检查、负载均衡、请求转发 | 不修改引擎内部调度，不管理单实例显存 |
| 内部 | 智能路由决策、指标采集 | 不生成 token，不写 CUDA kernel |



## 4. 量化验收指标

### 4.1 功能指标

| 指标 | 验收标准 |
|------|---------|
| OpenAI API 兼容性 | 使用官方 Python SDK 调用 `/v1/chat/completions` 能正常返回 |
| 流式响应 | SSE 流式输出正常，每 token 延迟 < 50ms（网关层开销） |
| 多后端路由 | 请求能按配置策略分发到不同后端实例 |
| 健康检查 | 后端实例下线后 3 秒内被摘除，恢复后 5 秒内重新加入 |
| 引擎适配 | 同一网关同时接入 vLLM 和 SGLang，请求均能正常处理 |

### 4.2 性能指标

| 指标 | 验收标准 | 测试条件 |
|------|---------|---------|
| 网关延迟开销 | P99 < 10ms | 不包含后端推理时间，纯调度转发 |
| 吞吐能力 | > 10k QPS | 单网关实例，8 核 CPU |
| 流式首包延迟 | P99 < 20ms | 从收到请求到转发第一个 token |

### 4.3 调度效果指标

| 指标 | 验收标准 |
|------|---------|
| 缓存感知路由命中率 | 相比随机/轮询策略，相同前缀请求的缓存命中率提升 > 30% |
| 负载均衡均匀度 | 各后端实例的请求数差异 < 20% |

### 4.4 可观测性指标

| 指标 | 验收标准 |
|------|---------|
| Prometheus 指标 | 暴露 QPS、延迟分布、缓存命中率、各实例负载 |
| 健康检查端点 | `/health` 返回 200，`/metrics` 返回 Prometheus 格式数据 |




## 5. 项目边界

### 5.1 本项目负责

- 请求路由与负载均衡
- 后端实例的健康检查与摘除
- vLLM / SGLang 的统一 API 抽象
- 缓存感知路由策略的实现
- Prometheus 指标的暴露

### 5.2 本项目不负责

**引擎层**
- 不修改 vLLM 或 SGLang 的内部调度逻辑
- 不实现 PagedAttention、RadixAttention 等显存管理算法
- 不写 CUDA kernel 或任何 GPU 相关代码

**推理层**
- 不实现模型的前向传播
- 不加载或管理模型权重
- 不生成 token 或处理 logits

**运维层**
- 不做后端实例的自动扩缩容（只提供指标建议）
- 不管理 GPU 显存的分配与回收
- 不实现模型的热加载或版本回滚

**业务层**
- 不做请求内容的深度校验（如 JSON Schema 验证）
- 不实现 API 鉴权与限流（可对接外部网关）
- 不实现请求的持久化或重放
