## 1. 背景与问题

### 1.1 现状
- 我们使用 vLLM / SGLang 部署 LLM 推理服务
- 单实例无法支撑高 QPS，需要多实例水平扩展

### 1.2 核心痛点
| 痛点 | 描述 |
|------|------|
| 负载均衡无效 | 传统 LB（Round-Robin / 最少连接）不理解 LLM 的 KV Cache 复用特性，相同前缀的请求被分散到不同实例，导致缓存命中率低、首 token 延迟高 |
| 引擎碎片化 | vLLM 和 SGLang 接口、行为不一致（如健康检查路径、metrics 格式、流式响应分隔符差异），运维需维护两套接入方案 |
| 运维黑盒 | 缺乏针对推理场景的监控指标（如 KV Cache 利用率、前缀匹配率、调度决策依据），无法判断路由策略是否生效 |

## 2. 项目目标

### 2.1 核心目标
- 实现 LLM-aware 的路由调度器（缓存感知 + 负载均衡）
- 统一 vLLM / SGLang 的 API 抽象，对外暴露 OpenAI 兼容接口
- 提供生产可用的可观测性（Prometheus 指标 + 健康检查）

### 2.2 非目标
- 不替换底层推理引擎（vLLM / SGLang）
- 不实现模型推理本身
- 不做 API 网关通用能力（鉴权、限流、持久化）
- 不做 GPU 资源管理或 CUDA 相关开发

## 3. 输入输出与系统边界

### 3.1 上边界：对外暴露的接口

- 暴露 OpenAI 兼容的 API（`/v1/chat/completions`、`/v1/completions`、`/v1/models`）
- 支持流式响应（SSE）
- 提供 `/health` 健康检查和 `/metrics` Prometheus 指标端点

### 3.2 下边界：接入的推理引擎

- 支持 vLLM 实例作为后端
- 支持 SGLang 实例作为后端
- 支持混合部署（同时接入两种引擎）

### 3.3 边界承诺

| 方向 | 做什么 | 不做什么 |
|------|--------|---------|
| 对上 | 提供 OpenAI 格式 API，透传标准参数 | 不做请求内容深度校验，不实现 OpenAI 高级功能校验 |
| 对下 | 健康检查、负载均衡、请求转发 | 不修改引擎内部调度，不管理单实例显存 |
| 内部 | 智能路由决策、指标采集 | 不生成 token，不写 CUDA kernel |

## 4. 技术栈选型

### 4.1 语言分工

| 组件 | 语言 | 理由 |

| 网关核心（路由、代理、健康检查） | **Go** | 高并发天然优势（goroutine + net/http），单机 10k+ QPS 无压力；编译为单二进制，部署简单 |

### 4.3 技术决策记录

| 决策 | 选择 | 备选 | 理由 |
|------|------|------|------|
| 网关语言 | Go | Python asyncio | Go 单机 10k QPS 有充分生产验证；Python asyncio 在高并发代理场景下 GIL 限制明显，实测通常 < 3k QPS |

| 配置方式 | YAML 文件 + 热加载 | etcd / Consul | 初期配置项少，YAML 简单直接；后续可扩展为动态配置 |

## 6. 缓存感知路由策略（详细设计）

### 6.1 问题定义

LLM 推理引擎利用 KV Cache 加速：已计算的 token 的 KV 对会缓存在 GPU 显存中。当新请求与已缓存请求共享前缀时，可直接复用 KV Cache，跳过前缀部分的计算，显著降低首 token 延迟（TTFT）。

**传统负载均衡（Round-Robin / 随机）的问题**：相同前缀的请求被分散到不同实例，每个实例都需要重新计算前缀，KV Cache 无法复用。

### 6.2 引擎缓存机制差异

| 维度 | vLLM Automatic Prefix Caching (APC) | SGLang RadixAttention |
|------|--------------------------------------|----------------------|
| 数据结构 | Hash 表（按 token 前缀 hash 索引） | Radix Tree（前缀树，自动合并共享前缀） |
| 缓存粒度 | 固定 block 大小（默认 16 token）的 KV Block | 任意长度的 token 前缀节点 |
| 匹配方式 | 精确前缀匹配（hash 查找） | 最长前缀匹配（radix tree 遍历） |
| 驱逐策略 | LRU（block 级别） | LRU（node 级别） |
| 感知能力 | 需外部路由保证前缀亲和性 | 内部可自动复用共享前缀，但仍受益于路由亲和 |

### 6.3 路由策略设计

**核心原则：将具有相同/相似前缀的请求路由到同一后端实例，最大化 KV Cache 复用。**

#### 策略一：前缀哈希路由（Prefix Hash Routing）

适用于 **vLLM** 和 **SGLang** 的基础策略。

```
输入：请求的 messages / prompt
  ↓
提取前缀：取 system_prompt + 前N个 user message（N 可配置）
  ↓
计算哈希：SHA256(prefix_tokens) → uint64
  ↓
取模映射：hash % len(healthy_backends) → 目标实例索引
  ↓
若目标实例负载过载（> 阈值）→ 降级为次优实例（一致性哈希顺时针下一节点）
```

- **前缀提取规则**：
  - Chat 请求：`system` message + 前 `prefix_depth` 轮 `user` message 的 token 序列
  - Completion 请求：prompt 前 `prefix_depth` 个 token
  - `prefix_depth` 默认值：256 token（约 1-2 轮对话），可按模型配置

- **一致性哈希**：使用一致性哈希环而非简单取模，避免后端实例变更时大面积缓存失效

#### 策略二：SGLang RadixAttention 感知路由（增强策略）

适用于 **SGLang** 后端，利用其 radix tree 的结构信息做更精准的路由。

- SGLang 的 `/metrics` 或专用端点暴露 radix tree 的前缀统计信息
- 网关定期拉取各 SGLang 实例的 radix tree 前缀分布
- 路由时：将请求前缀与各实例的 radix tree 做最长前缀匹配，选择匹配度最高的实例
- 降级：若无匹配前缀，回退到前缀哈希路由

**注意**：此策略依赖 SGLang 暴露 radix tree 信息，需确认 SGLang 版本是否支持。若不支持，SGLang 后端也使用策略一。

#### 策略三：负载均衡保底

当缓存感知路由导致负载严重不均时，触发负载均衡保底：

- 每个实例维护滑动窗口内的活跃请求数
- 若目标实例活跃请求数 > `max_load_factor` × 平均活跃请求数，则选择负载最低的实例
- `max_load_factor` 默认值：1.5（允许 50% 的不均衡以换取缓存命中率）

### 6.4 策略选择矩阵

| 后端类型 | 默认策略 | 增强策略 | 降级策略 |
|---------|---------|---------|---------|
| vLLM | 前缀哈希路由 | — | 负载均衡保底 |
| SGLang | 前缀哈希路由 | RadixAttention 感知路由（若可用） | 负载均衡保底 |
| 混合部署 | 按后端类型分别应用对应策略 | — | 负载均衡保底 |

### 6.5 缓存命中率度量

"缓存命中率提升 > 30%" 的精确定义：

- **度量方法**：在相同工作负载下，分别使用「缓存感知路由」和「Round-Robin 路由」运行，对比后端引擎报告的 KV Cache 命中率
- **数据来源**：
  - vLLM：`/metrics` 中的 `vllm:num_preempted`、`vllm:gpu_cache_usage_perc`
  - SGLang：`/metrics` 中的 `sglang_token_usage`、`sglang_cache_hit_rate`
- **计算方式**：`提升比例 = (策略A命中率 - RR命中率) / RR命中率 × 100%`
- **测试条件**：使用共享 system prompt + 多轮对话的工作负载，至少 4 个后端实例

## 7. 项目边界

### 7.1 本项目负责

- 请求路由与负载均衡
- 后端实例的健康检查与摘除
- vLLM / SGLang 的统一 API 抽象
- 缓存感知路由策略的实现（前缀哈希路由 + RadixAttention 感知路由）
- Prometheus 指标的暴露

### 7.2 本项目不负责

**引擎层**
- 不修改 vLLM 或 SGLang 的内部调度逻辑
- 不实现 PagedAttention、RadixAttention 等显存管理算法
- 不写 CUDA kernel 或任何 GPU 相关代码

**推理层**
- 不实现模型的前向传播
- 不加载或管理模型权重
- 不生成 token 或处理 logits

**运维层**
- 不做后端实例的自动扩缩容（只提供指标建议）
- 不管理 GPU 显存的分配与回收
- 不实现模型的热加载或版本回滚

**业务层**
- 不做请求内容的深度校验（如 JSON Schema 验证）
- 不实现 API 鉴权与限流（可对接外部网关）
- 不实现请求的持久化或重放
